# Deployment

## Recommended Server Setup

```bash
git clone https://github.com/CwWwWwW/alphaforge.git
cd alphaforge
python -m venv .venv
# Linux/macOS:
. .venv/bin/activate
# Windows PowerShell:
# .\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -r requirements.txt
playwright install chromium
```

## Configuration

```bash
cp config.example.json config.json
```

On Windows PowerShell:

```powershell
Copy-Item config.example.json config.json
```

Edit `config.json` with local authorized settings only. Do not commit it.

## Validate Before Running

```bash
python -m pytest
```

## Start the Main Workflow

```bash
python worldquant_auto_workflow.py
```

The legacy official workflow remains the production default path. Refactored, learning, replay, counterfactual, strategy, and observability features should remain shadow/advisory/read-only unless explicitly enabled and validated.

## Runtime Data

Do not add runtime state to git:

- `runtime/`
- `logs/`
- `runtime/db/`
- screenshots
- browser state
- private Alpha results
- credentials, cookies, or API keys

Before upgrading production deployments, back up `runtime/db` and any critical local logs outside the repository.

## Release Zip Deployment

GitHub Releases publish clean zip artifacts generated by `tools/build_release.py`. Release zips exclude runtime data, logs, databases, screenshots, secrets, and private results.

To deploy from a release zip, extract it to a clean directory, create a virtual environment, install requirements, copy `config.example.json` to `config.json`, and restore any private runtime backups only on the target machine.
